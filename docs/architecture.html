Data Flows

End-to-end flows split by business path. Each step is labeled by the primary integration style.

Ride Request & Dispatch

Rider creates a ride, Cab persists it, Matchmaking finds and reserves a driver, and Cab receives the final outcome.

1
Rider -> Gateway -> CabPOST /rides or routed cab API creates the ride request.
2
Cab -> PostgreSQLRide stored in cab domain with requested or matching state.
3
Cab -> KafkaPublishes ride-requested keyed by ride id.
4
Matchmaking -> LocationCalls Location base URL plus /internal/nearby for nearby online drivers.
5
Matchmaking -> Redis/PostgreSQLReserves driver with TTL and persists dispatch session plus assignment attempts.
6
Matchmaking -> KafkaPublishes driver-assigned or matchmaking-failed for Cab to consume.

Driver Response

Driver response APIs live in Cab Service; Matchmaking receives the result asynchronously and continues or completes dispatch.

1
Driver -> Gateway -> CabPOST /dispatch/driver-response with dispatch id, driver id, and accept/reject.
2
Cab -> KafkaPublishes assignment-accepted or assignment-rejected.
3
Matchmaking consumes responseAccepted completes assignment; rejected releases reservation and tries next candidate.
4
Reservation/session updateRedis reservation deleted or retained until assigned; PostgreSQL dispatch status moves forward.
5
Assignment outcomedriver-assigned updates Cab; matchmaking-failed marks no driver available.

Location & Realtime

Location Service owns current driver positions. Realtime Gateway is stateless and fans selected Kafka events to STOMP subscribers.

1
Driver -> Gateway -> Location/location/driver/online, /offline, and /update.
2
Location -> RedisMaintains drivers:geo, drivers:available, and drivers:available:geo.
3
Kafka realtime eventsRealtime listens to driver-location-events and assignment-requested.
4
Realtime -> RiderBroadcasts driver movement to /topic/trip/{rideId}.
5
Realtime -> DriverBroadcasts assignments to /topic/driver/{driverUserId}.

Auth & Onboarding

Authentication is synchronous in the current code path; user creation publishes downstream onboarding events.

1
Client -> Gateway -> Auth/auth/register, /auth/login, /auth/refresh, /auth/logout.
2
Auth -> UserFeign client calls /internal/users to create or fetch identity records.
3
Auth/User -> PostgreSQLCredentials, refresh tokens, and identity records are persisted separately.
4
User -> KafkaPublishes user.created after identity creation.
5
Driver consumes onboardingDriver Service consumes user.created for driver profile creation flow.

Services

Responsibility inventory based on current code/config, with storage and communication boundaries.

API Gateway

:8080REST edgeRedis rate limits

Routes by Eureka service id, validates JWT, applies rate limiting and circuit breakers, and checks internal API headers.

  • Configured prefixes: /auth/**, /users/**, /cab/**, /rides/**, /dispatch/**, /driver/**, /drivers/**, /matchmaking/**
  • Location routes: /location/driver/online, /location/driver/offline, /location/driver/update, guarded /location/internal/nearby
  • Routes now align with Cab, Driver, and Location controller paths.

Auth Service

:8091PostgreSQLJWT

Handles registration, login, refresh token lifecycle, logout, and user-service orchestration through internal REST calls.

User Service

:8081PostgreSQLKafka producer

Owns identity records and publishes user.created after successful user creation.

Cab Service

:8089PostgreSQLKafka in/out

Owns ride state, exposes rider ride APIs and driver response dispatch APIs, and consumes assignment results.

Driver Service

:8084PostgreSQLKafka consumer

Owns driver profile data and consumes user onboarding events. Availability/location realtime state is handled by Location Service in this codebase.

Location Service

:8090Redis GEOInternal lookup

Handles driver online/offline/update and nearby driver search using optimized online-driver GEO state.

  • /location/driver/online, /offline, /update
  • /location/internal/nearby

Matchmaking Service

:8087PostgreSQLRedis reservation

Consumes ride requests and driver responses, queries Location, reserves drivers with TTL, persists dispatch state, and emits assignment outcomes.

Realtime Gateway

:8095StatelessWebSocket/STOMP

Consumes realtime Kafka events and broadcasts to rider trip topics and driver assignment topics using Spring SimpleBroker.

Eureka Service

:8761Discovery

Service registry used by Gateway and services for discovery and load-balanced routing.

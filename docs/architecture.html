Detailed Internal Steps

🛫
1. Consume Event
Listen to ride-requested topic. Deserialize RideRequestedEvent containing: rideId, pickupLat, pickupLng, radius (default 5km).
📍
2. Call Location Service
GET /api/internal/drivers/nearby?lat=X&lng=Y&radius=5
Location Service returns list of driver IDs using Redis GEOSEARCH on drivers:available:geo key (v2 optimized).
✓
3. Filter Available Drivers
Check Redis SET drivers:available to verify driver is marked as available (not on ride). This eliminates offline drivers from geo results.
🔒
4. Reserve Driver (SETNX)
For each candidate driver (in order):
SETNX driver:reserved:{driverId} {rideId} EX 15
If success → proceed to assign. If fails → try next driver.
🚌
5. Call Driver Service
POST /api/drivers/{driverId}/assign
Body: { rideId, pickupAddress, eta }. Driver Service validates driver is still eligible, updates status to ASSIGNED in PostgreSQL.
🛫
6. Publish Result
On success: publish driver-assigned with {driverId, rideId, eta}.
On failure: publish matchmaking-failed with {rideId, reason}.
🛫
7. Handle Retry
Scheduler runs every 30s looking for rides stuck in REQUESTED status for >2min. Re-triggers matchmaking for unmatched rides.
🛫
8. Handle Rejection
Listen to assignment-rejected topic. Release Redis reservation, delete driver:reserved:{driverId}, retry with next driver.
🔄 Retry Logic on Driver Rejection/Timeout
Scenario: Driver doesn't respond within 15s or explicitly rejects
Action:
  1. Delete Redis key driver:reserved:{driverId} (release lock)
  2. Release driver in Location Service (mark available)
  3. Try next available driver from the list
  4. If no more drivers → publish matchmaking-failed

Redis Keys Used

drivers:available:geo - All driver locations
drivers:available - Available driver IDs SET
driver:reserved:{id} - 15s reservation lock
driver:location:{id} - Individual driver location

Complete Flow Summary

1. Rider → POST /api/rides → Cab Service → Kafka: ride-requested
2. Matchmaking ← Kafka: ride-requested
3. Matchmaking → Location Service → Redis (GEO search)
4. Matchmaking → Redis (SETNX 15s reserve driver)
5. Matchmaking → Driver Service (POST /assign)
6. Matchmaking → Kafka: driver-assigned
7. Cab Service ← Kafka: driver-assigned → Update DB
8. Driver → POST /dispatch/driver-response → Cab Service
9. Cab Service → Kafka: assignment-accepted/rejected
10. Matchmaking ← Kafka: assignment-rejected → Retry next driver
